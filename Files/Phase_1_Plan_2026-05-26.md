## Scope

Replace the `UserDefaults`-backed mock `AuthService` with a real Firebase Auth-backed implementation, keeping the public UI-facing API stable so `RootView`, `AuthView`, and `SettingsView` need only small touch-ups.

**Not in this phase:** Firestore push/pull (Phase 4), StoreKit, Cloud Functions, account deletion server-side cascade.

## Decisions Kenny needs to confirm before coding

1. **Anonymous auth vs. mandatory auth?**
   The current `AuthView` already has a `signInAsGuest()` mock that calls `mockSignIn(... provider: .mock)`. Recommendation: enable Firebase Anonymous Auth so a guest user still gets a stable `uid` for the local-first data layer to attach to. Easy to upgrade later via `link(with:)`.
2. **Email link vs. email/password?**
   Checklist says decision pending. Recommendation: email link — friendlier for indie users, no password reset flow.
3. **Account deletion behaviour for MVP?**
   Cloud Function cascade is App Store 5.1.1(v) required. Options:
   a. Wait until Phase 4 (Firestore exists) and ship deletion together.
   b. Phase 1 deletion only nukes Auth user + local SwiftData; Phase 4 adds Firestore cascade.
   Recommendation: (b). Reduces Phase 1 surface area.

## Code changes Claude will make in Phase 1

### 1. `Core/Services/AuthService.swift` — replace mock body, keep public API

Two changes:
- Introduce a private `AuthServicing` protocol-ish boundary (or keep concrete `AuthService` as a facade — Codex audit suggests the facade for speed; lean on that).
- Replace `UserDefaults` restore with a Firebase auth state listener:

  ```swift
  init() {
      handle = Auth.auth().addStateDidChangeListener { [weak self] _, user in
          self?.account = user.map { AccountSummary(from: $0) }
      }
  }
  deinit { Auth.auth().removeStateDidChangeListener(handle) }
  ```

- Implement `signInWithApple()` via `ASAuthorizationAppleIDProvider` + `OAuthCredential` exchange.
- Implement `signInWithEmail(_:)` as Firebase email link (`Auth.auth().sendSignInLink(toEmail:)` + URL handling in `DevCalApp`).
- Implement `signInAsGuest()` as `Auth.auth().signInAnonymously()`.
- Keep `signOut()` synchronous (`try Auth.auth().signOut()`).
- Add `deleteAccount()` that calls `Auth.auth().currentUser?.delete()` only (no Cloud Function cascade in Phase 1 per decision above). Wires into [SettingsView.swift:80](DevCal/DevCal/Features/Settings/SettingsView.swift#L80).
- Surface errors: convert `signIn*` to `async throws`; views handle via the same `systemAlert` pattern Phase 0 added.

### 2. `DevCalApp.swift` — bootstrap Firebase

- Add `import FirebaseCore` + `import FirebaseAuth` at top.
- In `init()`, before constructing `ModelContainer`: `FirebaseApp.configure()`.
- Email-link callback handler: register a `.onOpenURL` modifier on `RootView` that forwards to `Auth.auth().canHandle(url)` and then `signInWithEmail(link:)`.

### 3. `Features/Auth/AuthView.swift`

- Wrap each provider button's action in `Task { try await auth.signInWith*() }` with a try/catch → error alert. Currently calls are non-throwing `async`; updating to throws is a tiny ripple.
- Add a polished email-link "check your inbox" intermediate state.

### 4. `Features/Settings/SettingsView.swift:80` — wire account deletion

Replace the `TODO(firebase)` comment with `try await auth.deleteAccount()` inside a confirmation `systemAlert`. Add a copy line noting local data is also wiped (use `ModelContext.deleteAllData()` or iterate and delete via repositories — preferred to enqueue tombstones for Phase 4 sync).

### 5. `Features/Settings/CloudSyncSettingsView.swift:145` — leave alone

That's the "Sync now" button → Phase 4 hookup. Phase 1 still no-ops via `NoopSyncService`.

Check off the Phase 1 items as they land. Leave Phase 2+ items intact.

## Decisions Kenny needs to make before coding starts

### 1. Firestore region

Affects RTT for every read/write. Once a database is created in a region it cannot be moved — only copied. Options:

- **`asia-east1` (Taipei)** — best for Kenny + most likely indie user base in CJK markets.
- **`nam5` (US multi-region)** — Firebase default; good for global apps; mediocre latency from APAC.
- **`europe-west3` (Frankfurt)** — only relevant if EU is a priority.

Recommendation: `asia-east1` — match Kenny's locale, accept ~150ms penalty for North American users on the Free tier (the app is local-first, so reads/writes happen in the background).

### 2. Schema shape — flat vs nested sub-collections

Audit ([Architecture_Audit_2026-05-26.md](Architecture_Audit_2026-05-26.md) §"Phase 3: Sync Wiring") gestures at `users/{uid}/projects/...` etc. but does not commit.

- **Flat** — every collection lives at the root (`projects`, `transactions`, …) and every doc carries `ownerUid` + `projectId` fields. Pros: cross-project queries trivial; cleaner indexes; cheaper read patterns for "all my income across projects". Cons: security rules need explicit `ownerUid == request.auth.uid` checks on every collection.
- **Nested** — `users/{uid}/projects/{projectId}/transactions/{txnId}/…`. Pros: security rules trivial (`match /users/{uid}/{document=**}`); ownership is structural. Cons: cross-project aggregation needs a collection group query + composite indexes; deeper nesting hits the 100-level limit faster.

Recommendation: **flat with `ownerUid` filter** — DevCal's analytics screens already aggregate across projects (e.g. global income / category breakdowns), and a flat layout makes those single-collection queries instead of collection-group ones. Spend the budget on tighter rules.

### 3. Dev vs Prod Firebase project split

Today there is one project: `sideprofit-dev`. Once Phase 4 starts writing real Firestore data, schema migrations + Security Rules changes get destructive fast. Options:

- **Single project (`sideprofit-dev`) for the whole MVP** — fastest. Risk: any future user signs in to "production" against the same db dev was thrashed on.
- **Spin up `sideprofit-prod` now**, keep `sideprofit-dev` for active dev. Switch via build config or `Configuration.storekit`-style schema. Cost: re-download a second `GoogleService-Info.plist`, wire build configurations, keep two `Security Rules` files in lockstep.

Recommendation: **single project for the rest of the MVP coding loop**, schedule the `sideprofit-prod` split as a checklist item *just before App Store submission* (alongside the `com.kenny.DevCal` → real `com.sideprofit.*` bundle ID swap). Anything else doubles the manual-prereq surface for marginal benefit while there are zero real users.

### 4. MilestoneRepository — resolved in pre-flight

`MilestoneRepository` was added in pre-flight commit `8b5b61a`. Manual milestones sync via the same boundary as everything else. No further decision needed; leaving the bullet here so Kenny can see the cell is closed.

### 5. PendingSyncOperation persistence strategy

`NoopSyncService` currently keeps the last N ops in memory; if the user kills the app between a local write and the next `syncNow()`, the queue evaporates. Audit suggests persisting it. Options:

- **No persistence (current)** — accept that any unsynced op lost on cold launch. Sync would still recover via a full pull + last-write-wins on `updatedAt`. Simplest.
- **JSON file on disk** — write the queue to `Application Support/sync-queue.json` after each enqueue. Survives kill / reboot. Adds a file-write per local write but the volume is low.
- **SwiftData entity** — model the queue as a `PendingSyncOp` `@Model`. Most native, costs a schema migration and another migration vector.

Recommendation: **disk JSON**. The volume is "few writes per user gesture" and rebuilds the entire queue file is cheap (< 10ms). Avoids a SwiftData schema bump in the same phase that's already adding remote sync.

### Step 3 — Push pass

`syncNow()`'s push half:

1. Snapshot the queue.
2. For each entry, look up the Firestore doc path from `(kind, entityId, ownerUid)`.
3. Write the decoded DTO with `merge: true`.
4. Pop the entry on ack.
5. Surface errors as `SyncStatus.failed(...)` and leave the entry queued.

Schema mapping (flat, per decision 2):

```
collections:
  projects        - doc id = Project.id.uuidString
  transactions    - doc id = Transaction.id.uuidString (or .deterministicID for scheduler rows)
  timeLogs        - doc id = TimeLog.id.uuidString
  categoryItems   - doc id = CategoryItem.id.uuidString
  milestones      - doc id = Milestone.id.uuidString
  subscriptionStatus/current   - per-user, doc id = uid (separate root collection)

every doc carries:
  ownerUid: string
  isDeleted: bool
  updatedAt: Timestamp
```

Tests:

- Mock Firestore via a `RemoteWriting` protocol seam so unit tests don't need the emulator.
- `FirestorePushTests` — push success / network failure / partial batch.

Commit boundary: pushes land in Firestore for one entity kind at a time (start with `Project`, prove it, then expand).

**Step 3 — Project: DONE 2026-05-26.** Commit `3d29ab1`. Landed `runPush()`, the `RemoteWriting` protocol + `FirestoreRemoteWriter` production impl, ownerUid injection via `Firestore.Encoder`, and `firestore.rules` at the repo root (flat collections gated on `ownerUid == request.auth.uid`, `subscriptionStatus/{uid}` and `users/{uid}/deletionRequests/...` carved out for the Step 6 cascade function). Other entity kinds skip without ack until their commit teaches `encodePush` how to flatten them — the queue absorbs writes safely. Tests: `FirestoreSyncPushProjectTests` covers payload shape / skipped unimplemented kinds / queue-on-failure / no-op-when-disabled / tombstone round-trip.

**Step 3 — remaining entity kinds (Transaction, TimeLog, CategoryItem, Milestone):** next session. Each is a one-commit unit — extend `encodePush` plus add a focused test file.

### Step 4 — Pull pass

`syncNow()`'s pull half + on-launch pull:

1. For each collection, query `where("ownerUid", "==", uid).orderBy("updatedAt")` newest-first up to a paged limit.
2. Decode → reconcile with local SwiftData via last-write-wins on `updatedAt`.
3. Apply tombstones (`isDeleted == true`) by deleting the matching local row.

Conflict policy: last-write-wins, no merge, no user prompt. Documented as a known limitation for MVP.

Tests:

- `FirestorePullTests` — decode round-trip, tombstone apply, LWW resolution.

Commit boundary: end-to-end push + pull works for one entity. Repeat across the other five.

### Step 5 — `CloudSyncSettingsView` hookup

Replace the fake "Sync now" with `FirestoreSyncService.syncNow()`. Show:

Step 5 is small; bundle with the last entity-kind push/pull commit if it stays tight.

Today the local repos enqueue tombstones; the Cloud Function consumes the `users/{uid}/deletionRequests/...` doc + recursively deletes the user's docs. Out of scope for the local Swift loop — list the function source under `Functions/` once written, deploy via `firebase deploy --only functions`.
